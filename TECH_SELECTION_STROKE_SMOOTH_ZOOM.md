# 笔迹渲染“缩放无棱角”技术选型方案

## 1. 背景与目标

本项目当前渲染路径以“中心线点序列 + 压力”为输入，GPU 端在一次实例化绘制调用中生成笔身条带与端帽，并通过片元抗锯齿输出预乘透明度。

本方案文档聚焦的问题是：

- 在缩放（尤其是高倍放大）时，如何在“点是离散且数量固定/上限固定”的约束下，避免笔迹出现明显折线/棱角观感。
- 在满足“单次 glDrawArraysInstanced 绘制全部笔迹”“每条线独立颜色/透明度”的前提下，给出可落地的技术路线与取舍。

需要明确的物理事实：

- 如果“固定点”指的是“折线顶点（采样点）”，那么中心线本质上仍是折线；放大到足够大时必然能看出折线方向突变。能做的是把折线感推迟到更高倍率（让屏幕相邻点足够密、连接足够圆滑）。
- 如果“固定点”允许改为“固定曲线控制点（如贝塞尔控制点）”，则中心线是连续曲线；只要渲染阶段细分足够，就可以在任意倍率下保持曲线观感，不出现折线棱角。

## 2. 当前实现概览（作为选型基线）

- UI/交互：Kotlin + GLSurfaceView + ScaleGestureDetector
- 渲染：C++(NDK) + OpenGL ES 3.1 SSBO + Instanced Draw
- 数据：每条笔迹最多 1024 个点（超长笔迹 Kotlin 分段）；点坐标与压力进入 SSBO；元数据包含 baseWidth、RGBA、效果标记
- 绘制：每条笔迹一个 instance，顶点着色器用 gl_VertexID 生成整条笔迹的 TRIANGLE_STRIP（笔身 + 起/止端帽），片元着色器用 fwidth + smoothstep 做抗锯齿，输出预乘 alpha

对应架构描述详见：[TECH_ARCHITECTURE.md](TECH_ARCHITECTURE.md)

## 3. 问题定义：缩放时棱角从哪里来

棱角观感通常由两类原因叠加：

1) 中心线离散点导致的“方向突变”

- 顶点着色器主体几何依赖相邻点差分得到 dirPrev/dirNext；当点在屏幕上变稀疏（相邻点距离变大）时，方向变化在局部被放大，视觉上会更“折”。

2) LOD/限点导致的“屏幕弦长变大”

- 交互期间如果把每条笔迹参与渲染的最大点数显著降低（例如从 1024 降到 512），在高倍缩放时等价于“每段折线更长”，拐点更明显。

因此，“缩放无棱角”需要约束的核心量不是“世界空间点数”，而是“屏幕空间的相邻点间距/弦长”以及“连接处的视觉圆滑度”。

## 4. 候选方案

### 方案 A：保留“采样点序列”，用屏幕空间约束 + 圆角连接降低折线感（最小改造）

适用前提：

- 仍以点序列（positions[]）作为中心线输入，不改变 SSBO 数据结构。
- 允许在交互时调整 LOD 策略（不再硬阈值降点），并可增加拐角处视觉“圆角”处理。

关键策略：

1) LOD 以屏幕弦长为目标，而不是固定点上限

- 目标：保证绘制所用点序列在“当前缩放”下相邻点的屏幕距离不超过阈值（例如 1.0px~2.0px）。
- 做法：让“交互期 renderMaxPoints”随缩放倍率上调，而不是固定 512；在低倍缩小时才降点。

2) 连接处从“miter/bevel”提升为“round join”的视觉覆盖

- 当前主体条带连接主要靠 miter 并限制长度；它能避免尖刺，但无法把折线方向突变变成连续曲率变化。
- round join 的做法是对拐点处覆盖一个圆形/扇形的“圆角补片”，让方向变化呈现为圆滑过渡。
- 约束：保持单次 instanced draw 的情况下，round join 需要在同一个 instance 内额外输出 join 几何（或用片元 SDF 方式实现等价覆盖）。

优点：

- 不改数据结构，改动集中在“LOD 策略 + join 生成”。
- 与当前“单次实例化绘制调用”兼容性高。

缺点：

- 仍是折线中心线：在极高倍放大时无法从根上消除折线感，只能降低出现阈值。
- round join 的几何/片元覆盖会增加每条笔迹的顶点/片元负载，需要权衡预算。

结论：

- 作为短期可落地方案，用于显著改善“缩放过程中棱角明显”的用户体验。

### 方案 B：改为“固定曲线控制点”，在 shader 中连续求值并细分（根治折线感）

适用前提：

- 允许“固定的点”语义从“折线采样点”升级为“曲线控制点”（例如分段三次贝塞尔 4 控制点/段）。
- 允许在 SSBO 中存储每段的控制点与段索引信息，渲染阶段按 gl_VertexID 生成参数 t 并求值。

核心做法：

1) 数据表示：上传控制点而不是密集采样点

- 每段三次贝塞尔：P0、P1、P2、P3（世界坐标）+ 对应压力（可在控制点上存，或额外插值）。
- 还需要每条笔迹的“段范围/段数”元数据，便于 instance 内遍历。

2) 顶点着色器：连续求值中心线与切线

- 对每个渲染细分点：计算 P(t) 与 P'(t)，用 P'(t) 得到切线方向与法线，再按压力与宽度生成条带两侧。
- 细分数可以随缩放倍率/曲率估计动态调整：高倍放大时细分更多，保持平滑；低倍缩小时细分更少，保持性能。

优点：

- 从中心线定义上消除折线：高倍缩放下仍保持曲线观感，棱角问题根治。
- “固定点”数量可控：控制点数量远小于高精度采样点数量，长期看更适合大规模笔迹存储与无损缩放。

缺点：

- 数据结构、着色器与 CPU 侧拟合/分段策略都要升级，工程改造量较大。
- 需要谨慎设计“每条笔迹每段细分顶点数”的上限，避免 instance 内顶点数爆炸，影响单次 draw 的整体预算。

结论：

- 作为中长期高质量方案，解决“任意倍率无折线感”的本质需求。

## 5. 推荐路线（结合当前工程状态）

- 短期（先把体验问题压下去）：优先采用方案 A
  - 把“交互期固定 512 点”的策略改为“随缩放倍率动态调整的屏幕弦长策略”，避免高倍缩放时点过稀疏导致棱角立刻暴露。
  - 在需要进一步提升品质时，为拐点加入 round join 的补片/覆盖策略。

- 中长期（追求任意倍率无折线）：规划落地方案 B
  - 将 Kotlin 侧已存在的 Catmull-Rom→贝塞尔拟合产物从“采样点”升级为“控制点段”，并设计 SSBO 布局以支撑 instance 内遍历段与细分求值。

## 6. 与当前约束的匹配说明

- 单次 instanced draw：方案 A 与方案 B 都可以保持“每条笔迹一个 instance”，在单次 glDrawArraysInstanced 内完成全部绘制；差异仅在 instance 内如何生成顶点。
- 每条线独立颜色/透明度：两种方案都保持 meta 内 per-stroke RGBA 即可。
- 矢量缩放（宽度是否随缩放变化）：无论 A/B，笔宽是否保持“物理宽度不变”，都应在着色器端以 uViewScale 做乘/除来定义；该策略与“无棱角”是正交问题，可独立选择。
