# 技术设计（Technical Design）

## 总体方案
新增一个独立 Activity（简称“回放界面”），用于：
1) 解析用户输入的一段报点序列（带 eventType）  
2) 严格执行“双次拟合”处理链路  
3) 在一个自定义 View 内用 Canvas 绘制最终曲线，并提供原始点对照显示

该界面只依赖 Android Canvas，不依赖现有 OpenGL ES 渲染引擎，目的在于把“拟合算法与坐标变换”从渲染复杂度中解耦，便于快速定位“拐点被抹平/曲线偏离报点”的根因。

## 页面结构
- 入口：MainActivity 顶层叠加一个按钮，点击后 `startActivity` 进入回放界面
- 回放界面布局（建议用原生 View 体系实现，保持与现有工程一致）：
  - 数据源：直接写死一段固定报点 JSON（不提供输入与书写功能）
  - 操作区：显示开关（原始点散点）
  - 画布区：自定义 View（负责拟合结果与对照渲染）

## 数据模型
- `InputPoint`：
  - `x: Float`
  - `y: Float`
  - `timestamp: Long`
  - `eventType: EventType`
- `EventType`：`DOWN/MOVE/UP/CANCEL`（最小集合；扩展类型可映射为 MOVE 或忽略）
- `Stroke`：一次 `DOWN ... UP/CANCEL` 的点集合
- `QuadSegment`：`p0, p1, p2` 三点定义的二次贝塞尔段

## 处理流程拆解
### 1) 笔迹分段（按 eventType）
输入为一串点（可能包含多段笔迹）。处理时必须：
- 遇到 `DOWN` 开启新 stroke
- `MOVE` 追加到当前 stroke
- 遇到 `UP` 结束 stroke 并入队
- 遇到 `CANCEL` 丢弃或结束当前 stroke（以“结束”为默认）

### 2) 坐标归一化到宽度 1000px
设当前画布宽度为 `W`，归一化比例 `s = 1000 / W`，则：
- `x' = x * s`
- `y' = y * s`

归一化只依赖宽度，目的是使不同设备/窗口尺寸下拟合参数具有一致含义，避免“同一阈值在不同分辨率下表现不同”。

### 3) 第一次二次贝塞尔拟合（归一化坐标系）
核心要求：解决“曲线不经过实际报点、拐点被抹平”。

采用“插值式分段二次贝塞尔 + 角点强制分段”的策略：
- 端点约束：每一段曲线的端点 `P0/P2` 必须取自输入点（锚点），从而保证曲线经过锚点
- 角点保护：对输入点做角点检测，将角点强制作为段边界；角点处允许形成尖角而不是圆滑过渡
- 控制点计算：非角点处控制点 `P1` 根据相邻点方向生成，尽量连续；角点处可将 `P1` 贴近角点以降低圆滑程度

输出：
- `segments1: List<QuadSegment>`
- 以及用于后续流程的“拟合点序列 A”（通过对 `segments1` 重采样得到）

### 4) 从第一次拟合曲线重采样（得到点序列 A）
第二次拟合的输入需要一个点序列，因此需要重采样。关键约束：
- 重采样结果必须包含所有锚点（至少每段的端点与角点）
- 采样密度可用“固定步长（px）”或“固定每段采样数”实现，但必须可配置，便于复现与调参

### 5) 放大回屏幕坐标（得到点序列 B）
将点序列 A 放大回屏幕坐标系：
- `x = x' * (W / 1000)`
- `y = y' * (W / 1000)`

### 6) 第二次二次贝塞尔拟合（屏幕坐标系）
对点序列 B 使用与第一次相同的拟合策略，得到最终曲线 `segments2`。

为了避免“双次拟合”进一步抹平角点，第二次拟合必须延续：
- 锚点保留（端点、角点）
- 角点强制分段
- 控制点在角点处的“降低平滑”策略

## 绘制方案（Canvas）
在自定义 View 中：
- 将最终 `segments2` 转换为 `Path`：
  - `moveTo(P0)`
  - 对每段 `quadTo(P1, P2)`
- 以 `Paint.Style.STROKE` 绘制 Path
- 可选绘制对照：
  - 原始报点散点（小圆点）

## 可验证性
为确保“经过报点/保留拐点”的目标可回归：
- 以单元测试验证分段与角点检测的稳定性（同一输入输出锚点集合一致）
- 以简单几何断言验证：
  - 端点保持（DOWN/UP 对齐）
  - 重采样结果包含锚点（角点不可丢）
