# 提案：视频播放/会议共享屏幕场景下的手写渲染卡顿优化

## 背景（Context）
当前笔迹渲染链路以 Kotlin 输入 -> JNI -> C++/OpenGL ES 3.1 单次实例化绘制为主。在常规场景下可保持较好的流畅度，但当用户一边看视频（硬解 + 合成）或在在线会议中共享屏幕（录屏/编码 + 合成）时，系统的 CPU/GPU/内存带宽竞争显著加剧，导致以下现象更容易出现：

- 绘制帧出现明显掉帧/抖动（frame pacing 变差）
- 触控输入到画面反馈延迟增大（input-to-photon 变长）
- GLSurfaceView 渲染线程被系统调度挤压，出现长帧甚至“断续式刷新”

这类卡顿属于“外部高负载导致的渲染资源争抢”，目标不是让系统负载消失，而是让手写在高负载下具备更强的抗抖动能力，并提供可测量、可回归的优化闭环。

## 目标（Goals）
- 在视频播放/会议共享屏幕等高负载场景下，手写过程保持稳定可控的帧节奏，并显著降低“顿挫感”
- 降低输入到渲染反馈的尾延迟（尤其是 p95/p99），优先保障“跟手”
- 在不改变“单次 glDrawArraysInstanced 绘制全部笔迹”“每条线独立颜色/透明度”“无损矢量缩放”的核心约束下，提供自适应降级策略，使性能在负载波动时平滑过渡
- 引入可观测性（trace/指标/HUD），让问题可复现、可定位、可回归

## 非目标（Non-Goals）
- 不优化第三方视频/会议应用本身的性能与解码策略
- 不引入新的笔刷效果/材质体系（本提案聚焦于流畅性与调度）
- 不承诺所有设备在高负载下始终 60/120fps；目标是“稳定、可控、可感知改善”
