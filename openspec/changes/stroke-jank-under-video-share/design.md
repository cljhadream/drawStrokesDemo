# 技术设计（Technical Design）

## 总体思路
视频播放/共享屏幕开启时，系统通常会同时消耗：
- CPU：解码调度、会议编码、音视频处理、UI 逻辑
- GPU：视频纹理合成、UI 合成、后处理
- 内存带宽：视频帧搬运、合成读写

在这种竞争下，手写卡顿往往不是“单点性能瓶颈”，而是“帧节奏被打乱 + 同步点被放大”。因此设计目标是：
1) 让渲染节奏严格贴合 vsync，减少不必要的忙等与无效绘制  
2) 把“可能阻塞的 GPU 资源更新/同步”从关键路径移除或摊平  
3) 在负载波动时用“可逆的质量降级”换取稳定跟手  
4) 通过可观测性形成闭环：能定位到卡顿来自 CPU、GPU 还是 swap 阻塞

## 组件与职责
### 1) 指标与追踪（Observability）
- Kotlin 侧：在触控分发、JNI 调用、requestRender/vsync 回调处打点（用于 input-to-photon 归因）
- C++ 侧：在“输入队列消费、曲线拟合/切分、SSBO 更新、draw、swap”处打点
- 输出形态：
  - Perfetto/atrace 可抓取的 trace slice（用于离线分析）
  - 可选 HUD（用于现场复现与快速对比）

### 2) 帧调度（Frame Scheduler）
问题：GLSurfaceView 默认的连续渲染在高负载下容易形成“渲染线程忙等 + 交换阻塞抖动”，并且无法精确对齐 vsync。

方案：
- 采用 **按需渲染**：`RENDERMODE_WHEN_DIRTY`
- 由 **Choreographer vsync** 驱动每一帧的 `requestRender`（仅在需要更新时触发）
- 引入节流策略：
  - 书写中：尽量每个 vsync 都渲染
  - 非书写：仅在笔迹数据/视图变换变化时渲染
  - 高负载：允许自动降到每 2 个 vsync 渲染一次（稳定 30fps 的“均匀掉帧”优于随机抖动）

### 3) 输入与几何处理（Input + Geometry Pipeline）
目标：避免 UI 线程和 GL 线程互相阻塞，避免频繁分配与 O(N) 扫描。

方案：
- UI 线程只做轻量采样与坐标变换，写入 **无锁环形缓冲区**（或最小锁粒度队列）
- GL 线程/Native 线程按帧批量消费点，进行：
  - 增量贝塞尔拟合/切分（避免每帧对整条笔迹重算）
  - 增量更新 stroke 的包围盒/dirty region（为后续局部更新做准备）
- 所有临时对象复用，禁止在热路径中产生大量短命对象

### 4) GPU 数据更新与同步（Buffering Strategy）
高负载场景下最容易被放大的问题是 GPU/CPU 同步：
- `glBufferSubData`/`glMapBufferRange` 可能触发隐式等待
- swap 可能被合成/编码链路阻塞，导致 GL 线程“卡在末尾”

方案：
- 将“静态笔迹”和“动态笔迹（正在书写）”拆分：
  - 静态笔迹：尽量少更新，必要时批量重建
  - 动态笔迹：使用 **多缓冲（double/triple buffer）** 的 SSBO/VBO，按帧轮转，避免写入正在被 GPU 读取的缓冲
- 用 `glFenceSync` 做显式同步（仅在缓冲轮转追上 GPU 时等待），避免隐式 stall
- 对动态更新做合并：同一帧内对同一 stroke 的多次更新合并成一次上传

### 5) 自适应质量（Adaptive Quality）
原则：交互期优先“跟手与连续性”，视觉质量可短暂降低；负载缓解后自动恢复。

质量档位示例（从高到低）：
- Q3（高质量）：完整抗锯齿、round join/更高细分、完整压力曲线
- Q2（中质量）：降低细分密度、简化 join、缩小 AA 过渡带
- Q1（低质量/保命）：显著降低细分密度与片元成本，必要时将“在画中的笔迹”限制在一个轻量子通道

触发与滞回：
- 以近 1 秒窗口的长帧比例/连续掉帧段作为降级触发
- 恢复时需满足更严格条件，并设置冷却时间，避免频繁来回切换

可选（更强的抗抖动方案，代价更大）：
- **交互期临时预览层**：书写中将“最新增量笔迹”渲染到一个独立层（可为低成本路径），松手后用完整矢量路径回填重绘。矢量数据始终保留，缩放与最终清晰度不受影响。

### 6) 局部更新（Partial Update）
若设备与驱动支持：
- 使用 `eglSwapBuffersWithDamageKHR`（或等价机制）提交 dirty region，降低全屏带宽压力
- dirty region 来自“本帧新增笔迹包围盒 + 视图变换变化区域”

## 与现有架构的契合点
本设计在不改变核心渲染约束的前提下，主要增加：
- 帧调度与自适应质量模块（Kotlin 侧驱动 + Native 侧决策）
- 输入队列与增量几何处理（降低 UI/GL 线程互锁）
- GPU 多缓冲与显式同步策略（降低隐式 stall）

现有“单次实例化绘制”与“shader 中矢量缩放”保持不变，只调整“每帧上传多少数据、每帧生成多少几何、以及何时渲染”。
